# ADR-2026-04-19 — Verification artifact contract

## Status
- accepted
- date: 2026-04-19
- owner: ts-quality maintainers
- reviewers: operator, pi
- related_docs:
  - AGENTS.md
  - README.md
  - VERIFICATION.md
  - verification/verification.log
  - scripts/verify.mjs

## Executive summary

`ts-quality` keeps **`VERIFICATION.md` and `verification/verification.log` as checked-in generated reference artifacts**.

They are not ephemeral CI-only outputs, and they are not hand-authored narrative docs.
They are deterministic repo-local verification surfaces emitted by `scripts/verify.mjs` and reviewed in git.

This contract also keeps an important distinction explicit:

- the checked-in verification artifacts show the **latest recorded successful verification snapshot**
- they do **not** replace rerunning `npm run verify` or inspecting live CI/job status when current runtime truth matters

## Context

The repo already behaves as if these files are intentional review surfaces:

- `scripts/verify.mjs` writes both `VERIFICATION.md` and `verification/verification.log`
- `README.md` explicitly says the generated verification log is sanitized for volatile duration fields so it stays reviewable and stable across equivalent runs
- the sample-artifact stabilization pass already tightened `scripts/verify.mjs` to check idempotence and recorded a deferral for `task:191` because changing the checked-in status of these artifacts needed an explicit repo-level decision first

The unresolved question was whether these files should remain checked in as generated reference artifacts, or be treated as ephemeral outputs that only live in CI or local scratch space.

## Problem statement

Without an explicit contract, the repo can drift in two different ways:

1. keep checking in verification artifacts without agreeing why they exist or what truth they carry
2. untrack them for convenience and lose a useful reviewable record of the repo's current verification contract

The repo needs a durable answer about whether those artifacts are part of the reviewed surface.

## Decision drivers

- keep repo verification reviewable in normal code review
- preserve deterministic, stable artifacts instead of noisy raw transcripts
- avoid confusing checked-in verification records with live CI/runtime status
- align the artifact contract with the behavior already implemented in `scripts/verify.mjs`
- unblock `task:191` with an explicit contract choice

## Decision

`ts-quality` keeps the following verification artifact contract.

### 1) Artifact status

**`VERIFICATION.md` and `verification/verification.log` remain checked-in generated reference artifacts.**

That means:
- they are produced by `scripts/verify.mjs`
- they are committed and reviewed when relevant verification surfaces change
- they are not treated as ephemeral scratch output
- they are not hand-edited except through changing the generator or rerunning verification

### 2) Role of each file

**`VERIFICATION.md`** is the concise checked-in summary surface.
It records which verification commands succeeded and points at the detailed log.

**`verification/verification.log`** is the checked-in detailed execution record.
It keeps enough operator detail to review what verify actually ran, while still being normalized for stability.

### 3) Sanitization contract

The checked-in log is intentionally **sanitized for volatile duration noise**.
It is not meant to be a byte-for-byte raw shell transcript.
Equivalent successful runs should remain reviewable rather than churning solely because runtime durations changed.

### 4) Authority boundary

These files are **reference artifacts, not live authority**.

They show the latest recorded successful verification snapshot that was checked into the repo.
They do **not** replace:
- rerunning `npm run verify` when current correctness matters
- reading live CI/job state when current automation status matters
- the runtime validation commands defined in `AGENTS.md`

## Scope

- in scope:
  - `VERIFICATION.md`
  - `verification/verification.log`
  - the generator behavior in `scripts/verify.mjs`
  - the checked-in review contract for those artifacts
- out of scope:
  - changing the underlying verification command set itself
  - broader CI product decisions outside these repo-local artifacts
  - the projection-authority contract for `docs/project/*`, `next_session_prompt.md`, and `governance/work-items.json`

## Ownership / seam / policy notes

- owner: ts-quality maintainers
- allowed seams:
  - further stabilization of generated verification output
  - clearer rendering or structure for the checked-in verification summary/log
  - tighter validation that the checked-in artifacts stay generator-owned and deterministic
- prohibited patterns:
  - hand-editing `VERIFICATION.md` or `verification/verification.log` as prose artifacts
  - treating the checked-in verification files as a replacement for live verify/CI status
  - reverting to a noisy raw transcript format that makes equivalent runs churn constantly without an explicit superseding decision

## Alternatives considered

### Option A — Make both verification artifacts ephemeral outputs only
- description:
  - keep verification results only in CI logs or local working directories and stop checking them into git
- pros:
  - reduces checked-in artifact churn
  - avoids the risk of stale committed verification records being misread as current state
- cons:
  - removes a useful reviewed reference artifact from normal code review
  - hides the exact checked-in verification summary/log from the repo surface
  - conflicts with the repo's existing stabilization work, which already treats these outputs as intentional artifacts
- why not chosen:
  - the repo benefits from having these verification surfaces visible and reviewable in git

### Option B — Keep only VERIFICATION.md checked in and make the detailed log ephemeral
- description:
  - preserve a concise summary file but drop the checked-in detailed log
- pros:
  - reduces diff size
  - keeps some reviewable verification signal in git
- cons:
  - removes the detailed operator record that the summary points to
  - weakens reviewability when the command list stays the same but the detailed output matters
  - creates an awkward split surface with less evidence than the current generator already provides
- why not chosen:
  - the repo already has a stabilized detailed log, so dropping it would throw away useful evidence without solving the core authority question

### Option C — Keep both as checked-in generated reference artifacts with explicit non-authority boundaries
- description:
  - preserve the current generated files, keep them reviewed in git, and define them explicitly as reference artifacts rather than live authority
- pros:
  - matches current generator behavior
  - preserves reviewable verification evidence in the repo
  - keeps the detailed log available while allowing sanitization for stability
  - makes the authority boundary explicit instead of implicit
- cons:
  - still requires rerunning verify when relevant surfaces change
  - readers must understand the difference between recorded verification and live status
- why chosen or not chosen:
  - chosen because it best matches the repo's deterministic review posture and the generator already in use

## Consequences

### Positive
- verification status remains visible in normal code review
- the repo keeps a detailed generated log instead of only a terse summary
- artifact stabilization work stays aligned with the decision instead of being accidental precedent
- `task:191` can now optimize for stabilization under a chosen contract instead of deciding whether the files should exist at all

### Costs
- checked-in verification artifacts can still create diff noise when verification-relevant behavior changes legitimately
- maintainers must rerun verify when touching relevant surfaces
- readers need to understand that these files are snapshots, not live status APIs

### Risks
- a stale but checked-in verification snapshot may be mistaken for current repo truth
- future edits to `scripts/verify.mjs` could broaden churn if stabilization is not preserved
- teams may stop reviewing the log carefully if they assume the summary is always sufficient

### Mitigations
- keep README and this ADR explicit about the files being generated reference artifacts
- preserve sanitization of volatile duration fields and other non-semantic noise
- continue using `npm run verify` as the repo-root gate rather than treating committed verification files as self-justifying proof
- tighten generator-owned validation in follow-on work if hand edits or drift become a recurring problem

## Migration / rollout

- phase 1:
  - record this ADR
  - treat `task:191` as stabilization work under a checked-in-artifact contract rather than an open-ended keep-vs-delete choice
- phase 2:
  - preserve or improve generator stability while keeping the files reviewed in git
- rollback / escape hatch:
  - supersede this ADR if the repo later decides that verification artifacts should become ephemeral or be split differently

## Architecture fitness functions / validation

- invariant 1:
  - `scripts/verify.mjs` remains the generator for both checked-in verification artifacts
- invariant 2:
  - volatile timing noise is normalized so equivalent runs do not churn for incidental reasons alone
- invariant 3:
  - current correctness still comes from rerunning verify / checking CI, not from trusting stale checked-in snapshots blindly
- command checks:
  - `npm run verify`
  - `node ~/ai-society/core/agent-scripts/scripts/docs-list.mjs --docs . --strict`
- test / review gates:
  - changes to `scripts/verify.mjs` should keep `VERIFICATION.md` and `verification/verification.log` stable and intentional
  - if the repo wants to untrack or radically narrow these artifacts, record a superseding decision first
