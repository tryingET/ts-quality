# GitHub release draft — ts-quality v0.1.4

## Title

`ts-quality v0.1.4 — curated release pages and Trusted Publishing runtime repair`

## Release body

This release tightens the public release path after the `v0.1.3` GitHub Release proved the package but did not publish to npm. It keeps GitHub Release as the release intent, keeps npm publication in GitHub Actions, and makes the Trusted Publishing/OIDC path fail earlier with clearer evidence when external npm package settings are missing or mismatched.

### Fixed

- Published GitHub Release pages now use the release notes `## Title` for the release title and only the curated `## Release body` section for the public body, so draft scaffolding no longer leaks into the release page.
- The npm publish job now runs on Node `24`, satisfying npm Trusted Publishing's Node `>=22.14.0` runtime requirement while keeping the package's consumer engine at Node `>=20`.
- The release workflow now verifies npm `>=11.5.1` and GitHub OIDC request variables before `npm publish --provenance`, making repository-runtime problems distinct from npm-side Trusted Publisher configuration problems.

### Changed

- Release workflow docs now spell out the exact npm Trusted Publisher tuple: GitHub owner `tryingET`, repository `ts-quality`, workflow filename `release.yml`, and no environment unless one is intentionally added on both GitHub and npm.
- The changelog and release notes now follow the upstream Pi-style direction more closely: concise public-facing sections with user-visible effects first.

### Verification before release

```bash
npm run verify --silent
RELEASE_TAG=v0.1.4 GITHUB_REF_TYPE=tag npm run release:intent:check --silent
node ~/ai-society/core/agent-scripts/scripts/docs-list.mjs --docs . --strict
```
