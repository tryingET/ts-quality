# GitHub release draft — ts-quality v0.1.0

## Title

`ts-quality v0.1.0 — deterministic trust for TypeScript changes`

## Short description

`ts-quality` is an offline-first TypeScript quality platform for teams building with agents — and for agents building with agents.
It turns software-change evidence into explainable trust through mutation pressure, invariants, governance, and legitimacy checks.

## Release body

Today we are publishing the first public `ts-quality` alpha release.

This release now follows an explicit go decision after a successful staged-package rehearsal.
Before publishing for real, the repo proved the staged-package operator path through `npm publish --dry-run --access public` from `.ts-quality/npm/ts-quality/package`, found no publish-blocking defect, and kept the public release story aligned to that same operator path.

`ts-quality` is built for a very specific problem: modern TypeScript repos can generate plenty of code, but they still struggle to answer a harder question — **when is a change actually trustworthy?**

This project approaches that question deterministically.
Instead of confidence theater or opaque scoring, it combines explicit evidence layers into inspectable run artifacts:

- **Evidence** — CRAP + coverage for structural risk
- **Judgment** — deterministic mutation testing and merge-confidence scoring
- **Understanding** — invariant-driven expected behavior and missing-test obligations
- **Governance** — constitutional architectural boundaries, approvals, rollback evidence, and ownership constraints
- **Legitimacy** — agents, grants, attestations, overrides, and amendment flows

### Highlights in v0.1.0

- Public versioning starts at `0.1.0` to match the repo's alpha status instead of implying prior stable major generations
- Unified `ts-quality` CLI surface
- Stable evidence artifacts under `.ts-quality/runs/<run-id>/`
- Deterministic mutation execution with explicit execution fingerprints
- Focused invariant evidence instead of repo-global keyword coincidence
- Data-only config/support-module loading instead of executing project code
- Run-bound `controlPlane` snapshots so downstream decisions stay anchored to the exact reviewed run
- Governance hardening for dynamic imports, TS path aliases, extensionless imports, and tracked `require(...)` alias flow through destructuring, property access, rest bindings, and related container patterns
- Attestable authorization flows with run-bound evidence context
- Human-readable attestation verification attached to the evaluated run via `attestation-verify.txt`
- Amendment evaluation now persists both authoritative JSON and a concise human-readable summary
- Deterministic staged-package publish helpers via `npm run pack:ts-quality` and `npm run smoke:packaging`
- Fail-closed public-package contract checks for the staged manifest, staged file boundaries, and packed tarball file set
- Fresh-temp-project tarball install/load proof for the shipped CLI, API exports, and consumer type resolution
- Reviewed sample artifacts now include legitimacy-facing operator surfaces and stay deterministic under verification
- CI install flow now uses cached `npm ci` with retry-on-timeout behavior and avoids redundant reinstalls during verification

### Why this matters for agent-built systems

Agent-heavy codebases need more than linting and unit tests.
They need a durable way to answer:

- What exactly changed?
- Did tests apply meaningful pressure?
- Did the change violate a boundary through aliasing or indirection?
- Was an approval or attestation actually bound to this run?
- If the repo drifted after evaluation, do we fail closed?

That is the design center of `ts-quality`.

### First public package operator path

The first publish story is staged-package-first, and this release follows that now-explicit go path rather than a tentative publish plan.
`ts-quality` does **not** treat `packages/ts-quality/` as the publish root; instead it proves a publish-ready staged package, proves the packed tarball from that staged package, and only then publishes from the staged directory.

```bash
npm install
npm run build
npm run smoke:packaging
cd .ts-quality/npm/ts-quality/package
npm publish --access public
```

That proof path already verifies:

- the staged package manifest contract
- the staged package file-boundary contract
- the packed tarball file-set contract
- install/load behavior from a fresh temp project
- the shipped CLI surface, API exports, and consumer type resolution

### Repo-local evaluation quickstart

If you want to evaluate the project from source before or alongside the first public package publish:

```bash
npm install
npm run build
npm run verify
node dist/packages/ts-quality/src/cli.js check --root fixtures/governed-app
```

Then inspect:

```text
.ts-quality/runs/<run-id>/run.json
.ts-quality/runs/<run-id>/verdict.json
.ts-quality/runs/<run-id>/report.md
.ts-quality/runs/<run-id>/pr-summary.md
.ts-quality/runs/<run-id>/check-summary.txt
.ts-quality/runs/<run-id>/attestation-verify.txt
.ts-quality/runs/<run-id>/authorize.<agent>.<action>.json
.ts-quality/runs/<run-id>/bundle.<agent>.<action>.json
.ts-quality/amendments/<proposal-id>.result.json
.ts-quality/amendments/<proposal-id>.result.txt
```

### Legitimacy outputs operators can inspect

This release makes the legitimacy layer concrete instead of leaving it as generic policy language.
Operators can inspect:

- run-bound authorization decisions in `authorize.<agent>.<action>.json`
- the paired authorization bundle in `bundle.<agent>.<action>.json`
- human-readable attestation verification attached to the evaluated run in `attestation-verify.txt`
- amendment results in both authoritative JSON and concise reviewed text

If you want reviewed sample anchors before generating your own run, inspect:

```text
examples/artifacts/governed-app/authorize.release-bot.json
examples/artifacts/governed-app/authorize.maintainer-approved.json
examples/artifacts/governed-app/attestation.verify.txt
examples/artifacts/governed-app/amend.txt
```

Keep `run.json`, the authorization decision + bundle artifacts, and amendment JSON authoritative; the sample outputs are operator-facing projections, not a second legitimacy authority.

### Alpha status

`ts-quality` is currently **alpha** (`0.x.y`).
Before 1.0, breaking changes are allowed when they improve deterministic evidence, safety, trust-boundary correctness, or contract clarity.
Those changes should still be explicit in the changelog, reflected in docs, and backed by tests or validation.

One important current release detail: the public package path is intentionally a **staged package path**.
That is not accidental ceremony; it is how the repo keeps the published package truthful while the workspace build still targets the repo-root `dist/` tree.
