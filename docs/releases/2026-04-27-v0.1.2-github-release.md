# GitHub release draft — ts-quality v0.1.2

## Title

`ts-quality v0.1.2 — agent-facing adoption and operator guidance`

## Release body

This release improves the way AI agents and harnessed LLMs use `ts-quality` itself and roll it out into target repositories. It adds explicit guide routing, a one-slice external adoption walkthrough, and a machine-readable CLI command manifest so agents can choose the right docs and commands without guessing from scattered prose.

Publication still follows the repo's GitHub Release authority path. Publishing the GitHub Release for tag `v0.1.2` triggers `.github/workflows/release.yml`, which validates the tag/version contract, re-runs deterministic package proof, publishes the staged package to npm through Trusted Publishing/OIDC, verifies public installability, and attaches the proven tarball to the GitHub Release.

### Highlights

- Added `docs/harnessed-llm-operator-guide.md` for agents maintaining or reviewing the `ts-quality` repo itself.
- Added `docs/adoption/minimal-external-walkthrough.md`, a concrete target-repo adoption example with one behavior-bearing file, one invariant, one focused witness command, and one explicit run id.
- Added `docs/cli-command-manifest.json`, an authored machine-readable projection of CLI commands, allowed options, reads, writes, artifacts, and automation notes.
- Clarified brownfield vs greenfield adoption guidance so agents route to the right guide before creating config, witnesses, catalog entries, or runtime artifacts.
- Tightened adoption closure language around truthful evidence status, explicit `--changed` / `--run-id`, and routing blocking evidence debt before widening to another slice.

### Why it matters

`ts-quality` is intended for agent-built and agent-reviewed TypeScript systems. This release makes the operator layer itself more agent-readable:

- repo-maintenance agents get a repo-local read order, package map, validation matrix, and artifact-boundary guide;
- adoption agents get separate brownfield and greenfield playbooks instead of one ambiguous integration path;
- harnesses get a command manifest rather than having to scrape CLI help or infer command side effects;
- first-time adopters get a minimal one-slice walkthrough that avoids fake-green, repo-global rollout theater.

### Notes for alpha users

`ts-quality` remains alpha. The core runtime semantics in this release are unchanged; the release focuses on docs, adoption surfaces, and machine-readable operator metadata. Treat `docs/cli-command-manifest.json` as an authored projection of `packages/ts-quality/src/cli.ts`; future releases may generate or lint it directly from CLI contracts.
