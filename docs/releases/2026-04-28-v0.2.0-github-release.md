# GitHub release draft — ts-quality v0.2.0

## Title

`ts-quality v0.2.0 — adoption diagnostics and actionable evidence`

## Release body

This release is prepared through the repo's GitHub Release authority path. Publishing the GitHub Release for tag `v0.2.0` triggers `.github/workflows/publish.yml` in the `npm-publish` environment, which validates the tag/version contract, re-runs deterministic package proof, publishes the staged package to npm through Trusted Publishing/OIDC, verifies public installability, and attaches the proven tarball to the GitHub Release.

### Breaking Changes

- Run artifacts now declare schema version `0.2.0` instead of `0.1.0`. Agents, parsers, fixtures, dashboards, and prompts that hard-code `run.version === "0.1.0"` must accept `0.2.0` and tolerate the new additive fields (`coverageGeneration`, `analysisWarnings`, `mutationRemediation`, `verdict.confidenceBreakdown`, and `nextEvidenceAction`).

### Added

- Added configured LCOV generation sidecars (`coverage-generation.json` / `coverage-generation.txt`) whenever `coverage.generateCommand` is attempted.
- Added actionable surviving-mutant remediation details, including file, site id, span, original/replacement snippets, focused command context, and next-assertion hints.
- Added built-output LCOV warnings when changed `src/**/*.ts(x)` files are missing source coverage but LCOV covers `dist/**`, `lib/**`, `build/**`, or configured runtime mirror roots.
- Added deterministic merge-confidence breakdowns plus `next-evidence-action.json` / `.txt` so low confidence has an inspectable arithmetic trail and an immediate remediation target.
- Added read-only `ts-quality doctor` diagnostics plus `init --preset node-test`, `init --preset node-test-ts-dist`, and `init --preset vitest` starter config presets.

### Changed

- Bumped the run artifact schema version to `0.2.0` for the new adoption-readiness artifact surfaces.
- `check` now creates the parent directory for `coverage.lcovPath` before running configured `coverage.generateCommand`.
- Preserved the post-0.1.7 adoption fixes: configured LCOV generation during `check`, residual-pressure-specific verdict reasons, first-witness proof-command guidance, post-witness remaining-risk wording, and the pi-server LCOV rerun documentation.

### Fixed

- Fixed the pi-server-like adoption failure mode where LCOV generation could fail unless the target repo's coverage script created the destination directory itself.
- Fixed vague surviving-mutant output by surfacing the exact behavior location and assertion hint needed for the next remediation step.

### Agent migration notes

- Update any agent prompt, parser, or fixture that rejects run artifacts unless `version` is exactly `0.1.0`; it should accept `0.2.0` and continue reading older `0.1.0` artifacts where needed.
- Treat `mutationRemediation.survivors[]` and `nextEvidenceAction` as the preferred next-step surfaces before asking the operator for ad hoc interpretation of surviving mutants.
- When `analysisWarnings[].code` is `coverage-built-output-without-source-map`, advise the target repo to enable source-map coverage mapping, for example `NODE_OPTIONS=--enable-source-maps`, rather than claiming changed source is covered.
- If an adoption script invokes `coverage.generateCommand`, it no longer needs to pre-create the LCOV parent directory, but it must still write the configured LCOV file and fail truthfully on generation errors.
