# GitHub release draft — ts-quality v5.0.0

## Title

`ts-quality v5.0.0 — deterministic trust for TypeScript changes`

## Short description

`ts-quality` is an offline-first TypeScript quality platform for teams building with agents — and for agents building with agents.
It turns software-change evidence into explainable trust through mutation pressure, invariants, governance, and legitimacy checks.

## Release body

Today we are publishing the first public `ts-quality` repository.

`ts-quality` is built for a very specific problem: modern TypeScript repos can generate plenty of code, but they still struggle to answer a harder question — **when is a change actually trustworthy?**

This project approaches that question deterministically.
Instead of confidence theater or opaque scoring, it combines explicit evidence layers into inspectable run artifacts:

- **Evidence** — CRAP + coverage for structural risk
- **Judgment** — deterministic mutation testing and merge-confidence scoring
- **Understanding** — invariant-driven expected behavior and missing-test obligations
- **Governance** — constitutional architectural boundaries, approvals, rollback evidence, and ownership constraints
- **Legitimacy** — agents, grants, attestations, overrides, and amendment flows

### Highlights in v5.0.0

- Unified `ts-quality` CLI surface
- Stable evidence artifacts under `.ts-quality/runs/<run-id>/`
- Deterministic mutation execution with explicit execution fingerprints
- Focused invariant evidence instead of repo-global keyword coincidence
- Data-only config/support-module loading instead of executing project code
- Run-bound `controlPlane` snapshots so downstream decisions stay anchored to the exact reviewed run
- Governance hardening for dynamic imports, TS path aliases, extensionless imports, and tracked `require(...)` alias flow through destructuring, property access, rest bindings, and related container patterns
- Attestable authorization flows with run-bound evidence context
- Reviewed sample artifacts and repo verification designed to stay deterministic

### Why this matters for agent-built systems

Agent-heavy codebases need more than linting and unit tests.
They need a durable way to answer:

- What exactly changed?
- Did tests apply meaningful pressure?
- Did the change violate a boundary through aliasing or indirection?
- Was an approval or attestation actually bound to this run?
- If the repo drifted after evaluation, do we fail closed?

That is the design center of `ts-quality`.

### Quickstart

```bash
npm install
npm run build
npm run verify
node dist/packages/ts-quality/src/cli.js check --root fixtures/governed-app
```

Then inspect:

```text
.ts-quality/runs/<run-id>/run.json
.ts-quality/runs/<run-id>/verdict.json
.ts-quality/runs/<run-id>/report.md
.ts-quality/runs/<run-id>/pr-summary.md
```

### Alpha status

`ts-quality` is currently **alpha**.
Before 1.0, breaking changes are allowed when they improve deterministic evidence, safety, trust-boundary correctness, or contract clarity.
Those changes should still be explicit in the changelog, reflected in docs, and backed by tests or validation.
